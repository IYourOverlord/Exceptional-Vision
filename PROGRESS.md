## Этап 2 — LOD-структура и даунсемплинг (✅)

Без изменений с прошлой версии. Файлы:
`src/main/java/org/ex/exceptionalvision/world/lod/{PackedQuad,
NodeData, ColumnGrid, GreedyMesher, LodDownsampler, LevelStats,
LodBuildResult, MaterialPalette, LodNode, LodBuilder,
LodBuilderExecutor}.java`.

**Ключевые решения, обязательные к соблюдению в этапе 4/5/6** (см. также
javadoc `GreedyMesher`/`LodBuilder`):
1. Сетка узла всегда 16×16 ячеек; ячейка на уровне `L` = `2^L` блоков.
   `worldX = node.minX + localX * 2^lodLevel` — **шейдер этапа 4 уже
   реализует это** (`lod_quad.vert`), сверено построчно с `GreedyMesher.java`.
2. Конвенция width/height в `packed1` (верх: X/Z в ячейках; стены:
   вдоль границы в ячейках, по вертикали — в блоках) — **тоже уже
   перенесена в `lod_quad.vert`** построчно.
3. Стены только на внутренних границах узла — швы между узлами не
   закрыты (задача этапа 6).
4. Downsampling берёт максимум высоты, не среднее.
5. `LodNode.dirty` не подключён ни к чему автоматически (см. пробелы).

## Этап 3 — Дисковый кэш (✅, дополнено)

Файлы: `src/main/java/org/ex/exceptionalvision/cache/`.

**Изменения с прошлой версии этого файла:**
- `RegionCacheEntry` расширен полями `nodeIndexStart`/`nodeCount` —
  теперь `index.json` знает, какой диапазон записей в `nodes.bin`
  принадлежит региону. Старые файлы `index.json` без этих полей
  читаются нормально (дефолт 0/0) благодаря `CacheIndex.optionalInt`.
- `markRegionProcessed` **удалён**, вместо него:
    - `LodCacheWriter.appendRegion(...)` — первая обработка региона ИЛИ
      повторная обработка, при которой количество узлов изменилось
      (например, раньше несгенерированные чанки наконец сгенерировались).
      Сдвигает `quadOffset` каждого узла на текущий глобальный размер
      `quads.bin` (узлы из `LodBuildResult` приходят с
      **регион-локальными** смещениями — см. `LodBuilder.flatten`).
    - `LodCacheWriter.patchRegion(...)` — повторная обработка региона С
      ТЕМ ЖЕ количеством узлов (обычный случай: содержимое чанков
      изменилось, топология — нет). Патчит существующие записи узлов на
      месте, аппендит свежие квады (старые становятся мусором —
      подбираются `CacheCompactor`, как и раньше).
    - `LodCacheWriter.findRegionEntry(...)` — публичный метод для
      проверки, есть ли у региона уже присвоенный диапазон узлов (нужен
      `LodPipeline`, чтобы решить `appendRegion` vs `patchRegion`).
- **Известный пробел, специально задокументированный в javadoc
  `LodCacheWriter`:** компактится только `quads.bin`. Если у региона
  ПОСЛЕ `appendRegion` (не `patchRegion`) меняется количество узлов —
  старый диапазон узлов остаётся мёртвым грузом в `nodes.bin` навсегда
  (нет компактора для узлов). Редкий случай (только при изменении
  топологии чанков региона, не содержимого), но не забыт — не путать с
  "забыли", это осознанное решение по объёму работы, см. код.

Всё протестировано (см. «Как это тестировалось»), включая
специальный тест на **отсутствие дублирования узлов** при
`patchRegion`.

## Связка этапов 1→2→3 — `LodPipeline` (✅, новое)

Файл: `src/main/java/org/ex/exceptionalvision/pipeline/LodPipeline.java`.

Раньше в этом файле было явно написано: "`LodCacheWriter`/`LodBuilder`
пока не связаны друг с другом кодом". Это исправлено. `LodPipeline`:

1. При `start()` — загружает существующий кэш (`LodCacheLoader`),
   сканирует `regionDir`, ставит в очередь **только** регионы, для
   которых `LodCacheWriter.needsRecompute(...) == true` (низкий приоритет).
2. Запускает `WorldDirectoryWatcher` на `regionDir` — живые изменения
   получают более высокий приоритет, чем фоновый бэклог.
3. `RegionImportQueue` → `LodBuilderExecutor::submit` →
   `LodPipeline.onLodBuilt` → решение `appendRegion` vs `patchRegion`
   (через `findRegionEntry` + сравнение количества узлов) → запись в
   кэш → (опционально) колбэк `onRegionCached` наружу — это точка
   интеграции для GPU-буферов этапа 4 (см. ниже).

**Важное ограничение, специально описанное в javadoc класса:**
`LodPipeline.cachedAtStartup()` — это снимок **только на момент
старта**, он не обновляется при поступлении новых регионов (повторный
mmap растущего файла на каждый регион нецелесообразен через голый Java
NIO). Для живых обновлений GPU-буферов используйте `onRegionCached`.

**Не проверено юнит-тестом** (только компиляция против реальных
сигнатур `RegionImportQueue`/`WorldDirectoryWatcher` через
header-stub'ы — см. «Как это тестировалось»). Класс сам по себе тонкий
(только связывает уже протестированные части), но сквозной прогон
живого сценария (например, "регион появился → почему-то не тот путь
appendRegion/patchRegion выбрался") не прогонялся.

## Этап 4 — GPU-пайплайн, базовый (⚠️ подключён и прогнан в игре, 4 бага исправлены в этой сессии)

### Что реализовано

Шейдеры: `src/main/resources/assets/exceptional_vision/shaders/lod/`
- `quad_cull.comp` — compute-шейдер: frustum-culling (Gribb/Hartmann,
  positive-vertex тест) + выбор LOD-уровня по дистанции (простая
  политика "полосы дистанции", удваивающиеся по уровню — **временная
  заглушка**, реальный screen-space-error метод — это уже этап 5).
  Пишет compact-список видимых узлов + indirect draw commands +
  атомарный счётчик через `atomicAdd`.
- `lod_quad.vert` — процедурно генерирует геометрию квадов прямо из
  `QuadBuffer`/`NodeBuffer` SSBO (без vertex buffer вообще, через
  `gl_VertexID`/`gl_DrawID`). **Построчно сверен** с битовым layout
  `PackedQuad.java` и конвенцией width/height из `GreedyMesher.java`.
- `lod_quad.frag` — распаковывает цвет материала, сверен с
  `MaterialPalette.computeColor`'s packing.

Java: `src/main/java/org/ex/exceptionalvision/render/`
- `GpuCapabilities.java` — требует **OpenGL 4.6 core** (не 4.3!,
  см. ниже почему). Чистый fallback (`supported()==false`), без крашей.
- `ShaderProgram.java` — компиляция/линковка GLSL с classpath (НЕ через
  `ResourceManager` — не resource-pack-overridable, осознанное
  упрощение).
- `LodGpuBuffers.java` — владеет SSBO (node/quad/material/visible-index/
  indirect-commands/draw-counter). `upload(...)` льёт `LodCacheData`
  буферы напрямую в `glBufferData` — **без перепаковки** (для этого и
  проектировался формат этапа 3 — байт-в-байт совместим с GPU-структурой).
- `FrustumExtractor.java` — 6 плоскостей из view-projection матрицы
  через `Matrix4f.getRow` (специально не через `mCR`-поля JOML — там
  легко перепутать столбец/строку и молча сломать culling, что визуально
  выглядит как "часть мира просто не рисуется", а не как краш).
- `LodGpuPipeline.java` — последовательность кадра: сброс счётчика →
  compute dispatch → `glMemoryBarrier` → **явный readback** счётчика на
  CPU (это и есть "базовый" путь этапа 4 по roadmap, в отличие от
  полностью GPU-резидентного `glMultiDrawArraysIndirectCount` — это
  этап 5) → `glMultiDrawArraysIndirect` с явным count.
- `GlStateBackup.java` — сохраняет/восстанавливает program/VAO/
  depth-test/cull-face/blend вокруг нашего рисования (мы вклиниваемся в
  рендер ванильного Minecraft, нельзя оставлять GL-состояние грязным).
- `LodRenderManager.java` — точка входа в рендер.

Wiring: `src/main/java/org/ex/exceptionalvision/client/ExceptionalVisionClient.java`
- Регистрирует `LodRenderManager` на `NeoForge.EVENT_BUS`, вызывает
  `LodGpuPipeline.init()` в `FMLClientSetupEvent.enqueueWork`, создаёт
  `LodPipeline` на `LevelEvent.Load`/`Unload` (только singleplayer/LAN —
  `Minecraft.getSingleplayerServer() == null` → no-op, у клиента на
  удалённом сервере нет локальных файлов региона). Это тот самый wiring,
  который предыдущая версия этого файла ошибочно описывала как
  отсутствующий — см. «Уточнение» в шапке файла.

### Важное отклонение от спеки: НЕ Mixin, а `RenderLevelStageEvent`

`07_forge_integration.md` предлагает Mixin `@Inject` в хвост
`LevelRenderer#renderLevel`. **Я сознательно отклонился от этого** —
проверил через веб-поиск, что сигнатура этого приватного
обфусцированного метода менялась между версиями 1.21.x, и без
реальной сборки/маппингов угадывать её офлайн — рискованно (тихо
сломается, не будет ошибки компиляции). Вместо этого `LodRenderManager`
подписывается на официальное, документированное событие NeoForge
`RenderLevelStageEvent` (`Stage.AFTER_LEVEL`) — тот же момент кадра,
что и хвостовая инъекция, но через стабильный публичный API. Это
задокументировано в javadoc `LodRenderManager` подробно.

### Важное ограничение платформы: macOS

Требование **OpenGL 4.6 core** (не 4.3) — потому что вершинный шейдер
использует core `gl_DrawID` (без этого core-билтина пришлось бы тащить
`GL_ARB_shader_draw_parameters` и `#extension` прагму). Но macOS
устарел на OpenGL 4.1 (Apple задепрекейтил OpenGL целиком в пользу
Metal) — **на macOS весь этот пайплайн в принципе не заработает через
нативный OpenGL**, ни при каком ослаблении требований (compute-шейдеров
там нет вообще, начиная с версии 4.3). `GpuCapabilities.supported()`
корректно вернёт `false` на macOS — это ожидаемо, не баг.

## Сессия — первый реальный прогон в игре (4 бага найдены и исправлены)

Пользователь скомпилировал мод и запустил реальную игру — первый раз,
когда этап 4 вообще выполнялся вне компиляции против stub'ов. Ушли в
несколько итераций, каждая со своим симптомом и багом:

### Баг 1 — ничего не рисовалось вообще

**Симптом:** LOD-геометрия не появлялась нигде, никаких ошибок в логе.

**Причина:** двойное смещение в `lod_quad.vert`. `quad_cull.comp`
кодирует `node.quadOffset * 6u` в поле `first` indirect-команды; по
спецификации OpenGL `gl_VertexID` для non-indexed draw уже равен
`first + i`, то есть уже включает это смещение. Шейдер прибавлял
`node.quadOffset` **ещё раз** поверх — `quads[node.quadOffset +
quadIndex]` вместо `quads[quadIndex]`. Для первого узла в буфере
(`quadOffset == 0`) это случайно совпадало, поэтому не было ни ошибки
компиляции, ни краша — просто для всех остальных узлов чтение уходило
за пределы их диапазона квадов (часто и за пределы буфера — на
большинстве драйверов OOB-чтение из `std430` SSBO возвращает нули →
вырожденный квад нулевого размера → невидимая геометрия).

**Фикс:** `lod_quad.vert` — убрано повторное сложение
(`src/main/resources/assets/exceptional_vision/shaders/lod/lod_quad.vert`).

Заодно усилено (не точно доказанная причина, но обоснованное
укрепление): `LodGpuPipeline.runDrawPass` теперь явно выставляет
нужное GL-состояние (`glEnable(GL_DEPTH_TEST)` + `GL_LEQUAL` +
`glDepthMask(true)`, `glDisable(GL_BLEND)`, `glDisable(GL_CULL_FACE)`)
перед рисованием, вместо того чтобы наследовать что осталось от
предыдущего (полупрозрачного) прохода ванильного рендерера на
`Stage.AFTER_LEVEL` — наша процедурная геометрия не гарантирует
единой обмотки граней между верхней гранью узла и стеновыми гранями в
`lod_quad.vert`, так что оставленный culling мог невидимо резать часть
квадов в зависимости от направления взгляда.
`GlStateBackup` заодно научился сохранять/восстанавливать
`glDepthMask` (раньше не трогался вообще, хотя теперь мы его меняем).

### Баг 2 — геометрия появилась, но мерцала при повороте камеры

**Симптом:** после фикса №1 геометрия стала рисоваться, но мерцала —
пропадала и появлялась при повороте/наклоне камеры.

**Причина:** `quad_cull.comp` тестировал AABB узла в **абсолютных
мировых координатах** против плоскостей фрустума, извлечённых из
матрицы `projection * modelView`, где `modelView` — **только поворот,
без трансляции камеры** (так рендерит ванильный Minecraft — всё уже
camera-relative). Значит плоскости фрустума описывают фрустум с
вершиной в мировом origin (0,0,0), а не в реальной позиции камеры.
Проверка мирового AABB против такого фрустума случайно частично
совпадала только пока камера физически близко к origin мира (обычно
спавн) — при повороте то один, то другой узел проходил/не проходил
тест, отсюда и мерцание.

**Фикс:** `quad_cull.comp` — AABB переводится в camera-relative
координаты (`node.aabbMin.xyz - cameraWorldPos`) перед тестом, ровно
так же, как это уже делает `lod_quad.vert` для позиций вершин.

### Баг 3 — дистанция теперь стабильна, но цвета неверные и мерцание не пропало (даже стоя на месте, даже в прогруженных чанках)

**Симптом:** после фикса №2 мерцание от направления взгляда пропало,
но появилось статическое мерцание («будто одна текстура внутри
другой») — не зависящее от движения камеры, с явно неверными цветами
(лиловые/серые пятна), и затрагивающее **вообще всю сцену**, включая
честно прогруженные игроком чанки — не только удалённый LOD.

**Причина — не в рендере, а в данных.** Лог показал: регион
`x=-3, z=0` был обработан в 483 узла (нормально для частично
прогруженного пограничного региона) но **102 277 808 квадов** — у
всех соседних регионов с тем же числом узлов было 20-30 тысяч.

Корень: `GreedyMesher.wallBetween` считал стену между двумя соседними
ячейками, если хотя бы одна присутствует; отсутствующей стороне
присваивалась высота-«часовой» `Integer.MIN_VALUE`. Если реальная
(присутствующая) сторона имела **отрицательную** высоту (нормальная
ситуация ниже уровня моря при лимите высоты -64) —
`remaining = realHeight - Integer.MIN_VALUE` **не переполнялся**
обратно в безобидный отрицательный `int` (как было бы при
неотрицательном `realHeight`), а укладывался в честный положительный
`int` ≈ 2 147 483 584. `emitWallRun` затем послушно нарезал это на
~34 миллиона квадов высотой `MAX_WALL_QUAD_HEIGHT` (63) каждый — с
**одной-единственной границы ячеек** на краю прогруженной территории.

Дальше по цепочке: регион "успешно" (без исключений) дописался в общий
кэш → общий GPU SSBO с квадами раздулся до ~800+ МБ (минимальный
гарантированный спецификацией OpenGL лимит размера SSBO — всего
128 МБ) → буфер либо не влез, либо залился с обрезкой → поскольку это
**один общий буфер на всю карту**, битые данные сломали чтение
геометрии для всех узлов, не только для битого региона — отсюда и
статическое мерцание, и неверные цвета даже в честно прогруженных
чанках, и вероятная связь с последующими
`ClosedByInterruptException`/`RejectedExecutionException` в логе
(скорее всего последствие, не отдельная причина — тяжёлая обработка
раздутого региона могла спровоцировать прерывание пула потоков).

**Фикс:** `GreedyMesher.wallBetween` — если хотя бы одна из двух
соседних ячеек ещё не сгенерирована, стена теперь **не строится вообще**
(`if (!aPresent || !bPresent) return null;`), а не достраивается против
воображаемой высоты `Integer.MIN_VALUE`. Мы всё равно не знаем, что
там на самом деле — состыковка швов между узлами и так отдельно
отложена на этап 6, это согласуется с уже существующей заметкой в
javadoc класса. Плюс защитный потолок
`MAX_SANE_WALL_HEIGHT = 4096` в `emitWallRun` — на случай похожей
ошибки в будущем: деградация до "одна стена обрезана/неверна" вместо
"молча сгенерировать десятки миллионов квадов и убить общий GPU-буфер
для всей карты".

**Важно:** этот фикс не чинит уже испорченный дисковый кэш
задним числом — старые `nodes.bin`/`quads.bin` с этим мусором нужно
удалить (`<world>/lodcache/<dimension>/`), чтобы кэш пересобрался с
исправленным мешером.

2. **`LodNode.dirty` ни на что не влияет** — нет автоматического
   пересчёта, когда соседние чанки наконец генерируются.

6. `MaterialPalette` — без биомного тонирования травы/листвы.

   этапа 3 выше) — узлы, "осиротевшие" из-за `appendRegion` при смене

8. **Весь этап 4 не подключён и не проверен в реальном окружении**
   (см. раздел выше) — это НЕ то же самое, что "готово", несмотря на

9. Политика выбора LOD-уровня в compute-шейдере — временная заглушка
   (полосы дистанции), не финальная.

## Как это тестировалось

**Этапы 1–3 + `LodPipeline`:** офлайн, через самодельные, но
функционально корректные (не заглушки-пустышки) stub-классы Gson и
Minecraft API, плюс header-accurate stub'ы для `RegionImportQueue`/
`WorldDirectoryWatcher` (только публичные сигнатуры, не поведение).
Прогнаны юнит- и интеграционные тесты (не сохранены в репозитории —
одноразовые скрипты): геометрия greedy-меша, downsampling, полная
пирамида на тестовом регионе (1365 узлов), полный цикл кэша (запись →
загрузка → инкремент → компакция → инвалидация по версии формата),
**новое:** `appendRegion`/`patchRegion` с проверкой отсутствия
дублирования узлов при повторной обработке, и сквозной pipeline-тест
(симулированный регион → `LodBuilder` → `LodCacheWriter` →
`LodCacheLoader`, побайтовая проверка round-trip).

**Этап 4 (`render/` пакет):** ТОЛЬКО компиляция против вручную
написанных stub-классов `org.lwjgl.opengl.GL46C`,
`org.lwjgl.opengl.GL`/`GLCapabilities`, `org.lwjgl.system.MemoryUtil`,
`org.joml.{Matrix4f,Vector3f,Vector4f}`,
`net.neoforged.neoforge.client.event.RenderLevelStageEvent`,
`net.minecraft.client.Camera`/`net.minecraft.world.phys.Vec3` — все
сигнатуры в стабах писались ПО ПАМЯТИ и веб-поиску (для нескольких
ключевых: `BlockState#getMapColor`, структура `RenderLevelStageEvent`),
**не сверялись с реальными jar-файлами** (нет доступа к Maven Central
офлайн). Это качественно менее надёжная проверка, чем для этапов 1–3 —
там stub'ы были дополнены исполняемыми тестами с проверкой поведения
(не только компиляции), а этап 4 — только компиляция.

**Первым делом при работе в среде с доступом к интернету/Gradle** —
прогнать `./gradlew build` и `./gradlew runClient`, начиная с пакета
`render/` (самый непроверенный код) и `LodRenderManager`'а
(NeoForge event API мог измениться между версиями).

Если захочешь добавить настоящий JUnit — в `build.gradle` пока нет
тестового source set (`src/test/java` не настроен).
