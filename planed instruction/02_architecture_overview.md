# Общая архитектура

## Цель системы

Отрисовывать местность на дистанциях, значительно превышающих обычный
render distance Minecraft, используя упрощённую (LOD) геометрию,
построенную из **уже сохранённых на диске данных мира** — включая чанки,
сгенерированные заранее сторонними инструментами (Chunky и подобные),
без необходимости, чтобы игрок посещал эти чанки лично.

## Высокоуровневые компоненты

1. **World Data Reader** (`05_world_data_ingestion.md`)
   Независимый от игровой сессии читатель `region/*.mca`. Работает в
   фоновом потоке, следит за изменениями файлов на диске (новые регионы,
   изменённые чанки).

2. **LOD Builder** (`03_data_formats.md`)
   Принимает сырые данные чанков, строит иерархию LOD (quadtree по X/Z),
   даунсемплит колонки блоков в компактное представление, выполняет
   greedy meshing → генерирует упакованные "квады".

3. **Disk Cache** (`06_disk_cache_format.md`)
   Персистентное хранилище уже посчитанных LOD-данных, чтобы не
   пересчитывать их при каждом запуске игры. Инкрементально обновляется.

4. **GPU Data Manager** (`04_gpu_pipeline.md`)
   Java-слой, управляющий SSBO-буферами: загрузка новых квадов через
   persistent mapped buffers, обновление метаданных узлов quadtree.

5. **GPU Renderer** (`04_gpu_pipeline.md`)
   Compute shader для culling + indirect multidraw + vertex shader,
   генерирующий геометрию квадов на лету. CPU почти не участвует в кадре.

6. **Forge Integration Layer** (`07_forge_integration.md`)
   Mixin-хуки в `LevelRenderer`, отдельный рендер-проход, не
   конфликтующий с ванильным чанковым рендером (нужным для ближней
   дистанции).

## Поток данных (сквозной путь)

```
region/*.mca (диск, включая Chunky-регионы)
        │
        ▼
World Data Reader (фоновый I/O поток)
        │  сырые блоки чанка
        ▼
LOD Builder (пул потоков даунсемплинга)
        │  упакованные квады + AABB узла
        ▼
Disk Cache  ──────────────┐  (сохранить для следующего запуска)
        │                 │
        ▼                 ▼
GPU Data Manager  (SSBO: QuadBuffer, NodeBuffer)
        │
        ▼
GPU Renderer (compute culling → indirect multidraw → vertex shader)
        │
        ▼
Кадр: ближний ванильный рендер + дальний LOD рендер, состыкованные на границе
```

## Инварианты системы

- CPU никогда не строит вершинную геометрию LOD впрямую — только
  компактные упакованные данные (см. `03_data_formats.md`).
- Frustum culling и решение "рисовать ли узел" принимается на GPU в
  compute shader, не в Java-коде каждый кадр.
- Чтение мира полностью отделено от игровой логики загрузки чанков —
  никакой зависимости от того, находится ли чанк в радиусе игрока.
- Все LOD-данные — временный производный кэш; исходные `.mca` файлы
  никогда не изменяются модом.

## Требования к окружению

- Minecraft 1.21.1, Forge (MDK текущей версии под 1.21.1), Java 21.
- LWJGL (поставляется с Minecraft) для прямых вызовов OpenGL.
- OpenGL 4.3 минимум (compute shaders), OpenGL 4.6 предпочтительно
  (`ARB_indirect_parameters` для отрисовки без чтения счётчика на CPU).
