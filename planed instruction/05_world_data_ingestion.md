# Чтение данных мира независимо от игровой сессии

## Цель

Строить LOD-данные из **всех уже сгенерированных регионов на диске**,
а не только из чанков, загруженных текущей игровой сессией. Это
позволяет отрисовывать территорию, заранее сгенерированную сторонними
инструментами (например Chunky), даже если игрок физически там не был.

## Расположение файлов

```
<world>/region/r.X.Z.mca            — overworld
<world>/DIM-1/region/r.X.Z.mca      — нижний мир
<world>/DIM1/region/r.X.Z.mca       — край
<world>/dimensions/<namespace>/<dim>/region/r.X.Z.mca  — кастомные измерения (1.16+)
```

Формат `.mca` (Anvil) — публично задокументированная спецификация:
заголовок из 1024 записей (смещение + размер сектора) + 1024 таймстампа,
далее сжатые (обычно zlib) NBT-чанки по 4096-байтным секторам.

## Архитектура ридера

```java
public final class RegionFileReader {
    // Чтение заголовка региона: таблица смещений секторов и таймстампов
    public RegionHeader readHeader(Path mcaFile) throws IOException;

    // Чтение и распаковка одного чанка по локальным координатам 0..31
    public CompoundTag readChunk(Path mcaFile, int localX, int localZ) throws IOException;

    // Извлечение только той информации, что нужна для LOD:
    // высоты колонок + материал верхнего непрозрачного блока на каждую (x,z)
    public ChunkHeightData extractHeightData(CompoundTag chunkNbt);
}
```

Важно: для LOD не нужен полный `Level`/`ChunkAccess` со всей игровой
логикой (entity, block entities, тикающие данные) — читается только
секция `sections` с `block_states` (палитра + индексы), что резко
снижает объём обрабатываемых данных по сравнению с полной загрузкой
чанка через игровой движок.

## Фоновый мониторинг директории

```java
public final class WorldDirectoryWatcher {
    private final WatchService watchService;

    public void watch(Path regionDir, Consumer<Path> onRegionChanged) {
        // WatchService на ENTRY_CREATE / ENTRY_MODIFY
        // Дебаунс: Chunky пишет регион пакетами, ждать затухания
        // изменений (например 2 секунды без новых событий) перед
        // тем как ставить регион в очередь на обработку LOD-builder'ом.
    }
}
```

Это позволяет: если Chunky догенерировал новый регион во время открытой
игры — мод подхватывает его без перезапуска клиента.

## Многопоточная организация

- Отдельный `ExecutorService` (например `Executors.newFixedThreadPool(2)`)
  только для файлового I/O — чтение .mca не должно блокировать поток
  даунсемплинга LOD и тем более рендер-поток.
- Очередь задач по приоритету: регионы ближе к игроку обрабатываются
  первыми, дальние — в фоне с низким приоритетом.
- Result future из чтения региона передаётся в LOD Builder
  (`03_data_formats.md`) как источник сырых данных колонок.

## Обработка команды импорта (аналог `/voxy import world`)

Не заимствование конкретной реализации, а независимая функция с той же
идеей — команда, которая ставит в очередь **все** существующие регионы
мира на построение LOD, а не только те, что попадут в радиус игрока
со временем:

```java
public void queueFullWorldImport(Path worldDir) {
    Path regionDir = worldDir.resolve("region");
    try (Stream<Path> files = Files.list(regionDir)) {
        files.filter(p -> p.toString().endsWith(".mca"))
             .forEach(regionQueue::submitLowPriority);
    }
}
```

## Особые случаи

- **Пустые/повреждённые сектора региона** — .mca может содержать
  "дыры" (chunk status < `full`, например только сгенерированный
  рельеф без структур). Такие чанки нужно либо пропускать, либо строить
  LOD только из доступных секций, помечая узел как "неполный" для
  последующего дозаполнения.
- **Версии формата чанка.** Начиная с 1.18 высота мира расширена
  (-64..320), секции идентифицируются по `Y` от -4 до 19 — учитывать
  при вычислении битовых полей высоты в формате квада (см.
  `03_data_formats.md`, 12 бит на Y достаточно для полного диапазона).
